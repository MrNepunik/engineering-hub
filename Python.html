Python & the Abstract Machine

// bytecode, GIL, and duck typing

01 The Visionary

Guido van Rossum

Released in 1991, Python was designed as a successor to the ABC language. Van Rossum's "Pythonic" philosophy: "one—and preferably only one—obvious way to do it." This focus on clean implementation logic allowed Python to dominate academic and AI fields.

02 How Python Executes

Python doesn't run source code directly. It follows a pipeline:

  1. Compilation: Source (.py) → Bytecode (.pyc)
  2. Virtual Machine: Bytecode is shipped to the Python Virtual Machine (PVM)
  3. Runtime: The PVM executes instructions

The Bytecode middle step makes Python platform-independent — the PVM handles hardware translation.

03 Memory Management & Reference Counting

Python manages memory via Reference Counting + a Cyclic Garbage Collector. Every object has a counter. When you assign a variable, the counter increments. When you delete it, it decrements. When it hits zero, memory is instantly reclaimed.

04 The Global Interpreter Lock (GIL)

The GIL is a mutex that prevents multiple native threads from executing Python bytecodes simultaneously. Even on a 16-core CPU, a single Python process can only use one core at a time. Engineers bypass this with Multiprocessing or AsyncIO.

05 Dynamic Typing & Duck Typing

"If it walks like a duck and quacks like a duck, it's a duck." You don't define types — the PVM checks if an object has the required methods at runtime. Extreme flexibility, but requires strict unit testing.
